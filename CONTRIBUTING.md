# Contributing To Gridflow

Thanks for helping improve Gridflow.

## Before You Start

- Do not commit personal task data, exported JSON files, recovered imports, DMGs, or local build output.
- If you add screenshots, sample exports, or demo content, sanitize everything first.

## Local Setup

```bash
cd Gridflow
export SWIFTPM_MODULECACHE_OVERRIDE="$PWD/.build/module-cache"
export CLANG_MODULE_CACHE_PATH="$PWD/.build/clang-module-cache"
swift build --disable-sandbox --scratch-path "$PWD/.build/swiftpm" -c release
```

To run the app locally:

```bash
cd Gridflow
./run.sh
```

To create a local DMG:

```bash
cd Gridflow
APP_VERSION=1.0.0 APP_BUILD_NUMBER=1 ./scripts/build_dmg.sh
```

## Pull Requests

- Keep each PR focused on one change set.
- Describe the user-facing impact and include manual test steps.
- Add screenshots for UI changes, but only with demo data.
- Update docs when setup, release behavior, or contributor workflow changes.
- Avoid renaming historical internal paths unless the PR is explicitly about repository restructuring.

## Quality Checks

Before opening a PR, please run:

```bash
cd Gridflow
swift build --disable-sandbox --scratch-path "$PWD/.build/swiftpm" -c release
```

CI is currently build-focused. If you add tests, wire them into the existing workflow.

## Privacy Checklist

Before pushing, make sure your diff does not contain:

- `~/Library/Application Support/Gridflow/storage.json`
- Personal screenshots or exported task lists
- Files such as `Gridflow-export.json`, recovered imports, `.dmg`, `.app`, or `.zip`
- Build directories such as `.build/`, `dist/`, or derived data output

## Release Flow

1. Confirm README, screenshots, and docs are up to date.
2. Run a local release build if you want a smoke test before tagging.
3. Create an annotated tag such as `v1.0.0`.
4. Push the tag to GitHub.
5. Verify that the GitHub Release contains the DMG, checksum, and autogenerated source archives.
